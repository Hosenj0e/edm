# 1 Общая (теоретическая) часть

## 1.1 Постановка задачи

### 1.1.1 Обоснование необходимости разработки

В соответствии с заданием на выпускную квалификационную работу, требуется разработать информационную систему управления документооборотом учебных ведомостей в политехническом колледже НовГУ имени Ярослава Мудрого.

**Целевая аудитория системы:**

- Сотрудники учебной части — формирование ведомостей, контроль успеваемости, подготовка отчётов.
- Преподаватели — ввод текущих и экзаменационных оценок, создание ведомостей, отправка на согласование.
- Заместитель директора по УМ и ВР — согласование (подтверждение) ведомостей, контроль процесса.
- Администратор системы — управление учётными записями пользователей, полный доступ ко всем функциям.

**Существующие проблемы:**

- Высокие временные затраты на ручное заполнение бумажных ведомостей — преподаватель тратит в среднем 15–20 минут на заполнение одной ведомости вручную.
- Дублирование данных — одна и та же информация вносится в бумажную ведомость, затем в Excel-таблицу, затем передаётся в учебную часть.
- Риск потери и порчи бумажных ведомостей — отсутствие резервных копий.
- Отсутствие разграничения прав доступа — любой сотрудник может внести изменения в бумажный документ.
- Невозможность оперативного контроля — заместитель директора не может в реальном времени отслеживать статус заполнения ведомостей.
- Отсутствие единого формата — разные преподаватели заполняют ведомости по-разному, что затрудняет обработку.
- Невозможность работы при отсутствии интернета — если система будет только веб-ориентированной, при отключении сети работа остановится.

**Что даст разработка:**

- Сокращение времени на заполнение ведомости с 15–20 минут до 3–5 минут (за счёт автоматического формирования списка студентов).
- Единый формат ведомости — система генерирует документ по утверждённому образцу колледжа.
- Достоверность и непротиворечивость информации — данные вводятся один раз и используются повторно.
- Разграничение прав доступа — три роли (администратор, преподаватель, заместитель директора) с различным набором функций.
- Процесс согласования — преподаватель отправляет ведомость на согласование, заместитель директора подтверждает или отклоняет.
- Возможность работы в офлайн-режиме — приложение полностью функционально без подключения к интернету.
- Экспорт и печать — ведомость можно экспортировать в HTML и распечатать через стандартный браузер.
- Защита данных — пароли хешируются алгоритмом PBKDF2-SHA256 (120000 итераций).

### 1.1.2 Технико-математическое описание задачи

**Математические методы:**

- Статистическая обработка данных — подсчёт количества оценок каждого типа (5, 4, 3, 2) в ведомости для формирования итоговой строки.
- Криптографические методы — алгоритм PBKDF2-SHA256 для безопасного хранения паролей пользователей. Параметры: 120000 итераций, соль 16 байт, длина производного ключа 32 байта. Используется функция сравнения за постоянное время (constant-time comparison) для защиты от timing-атак.
- Алгоритмы фильтрации и поиска — фильтрация групп по наличию ведомостей, по наличию студентов; поиск по номеру группы (подстрока, регистронезависимый).
- Генерация уникальных идентификаторов — UUID v4 для первичных ключей всех сущностей (группы, студенты, дисциплины, ведомости).

**Технологии разработки:**

- JSON — формат хранения данных синхронизации (подготовка к будущей серверной части).
- UUID — формат уникальных идентификаторов записей.

### 1.1.3 Характеристика бизнес-процессов

**Анализ предметной области:**

Процесс документооборота учебных ведомостей в колледже включает следующие этапы:

1. Формирование списка студентов группы (учебная часть).
2. Создание бланка ведомости по дисциплине (учебная часть).
3. Заполнение оценок преподавателем.
4. Передача заполненной ведомости в учебную часть.
5. Проверка и согласование заместителем директора.
6. Архивирование ведомости.

**Группы пользователей и их функции:**

Таблица 1.2 — Роли пользователей и их функции

| Роль | Функции |
|------|---------|
| Администратор | Создание/удаление учётных записей; назначение ролей; создание/удаление групп; все функции преподавателя и зам. директора |
| Преподаватель | Просмотр групп и студентов; создание ведомостей; выставление оценок; редактирование студентов в ведомости; отправка ведомости на согласование |
| Заместитель директора | Просмотр групп и студентов; создание групп и ведомостей; выставление оценок; согласование (подтверждение) ведомостей; отклонение ведомостей; экспорт и печать ведомостей |

Преподаватель          Учебная часть         Зам. директора

Преимущества TO-BE: автоматическое формирование списка, единый формат, мгновенное согласование, экспорт для печати.

## 1.2 Обоснование проектных решений

### 1.2.1 Обоснование выбора языков программирования

**C++20** выбран в качестве основного языка программирования по следующим причинам:

1. *Высокая производительность* — компилируемый язык обеспечивает минимальное потребление ресурсов и быстрый отклик интерфейса.
2. *Прямой доступ к системным API* — возможность использования нативных функций ОС для мониторинга сети, работы с файловой системой.
3. *Интеграция с Qt* — фреймворк Qt написан на C++ и предоставляет наиболее полный API именно для этого языка.
4. *Криптографические операции* — возможность низкоуровневой оптимизации алгоритмов хеширования.
5. *Стандарт C++20* — использование современных возможностей языка (structured bindings, std::move, constexpr).

**QML (Qt Quick Markup Language)** выбран для разработки пользовательского интерфейса:

1. *Декларативный подход* — описание интерфейса в виде дерева компонентов, что упрощает разработку и поддержку.
2. *Разделение логики и представления* — бизнес-логика на C++, отображение на QML.
3. *Адаптивность* — автоматическое масштабирование элементов под разные разрешения экрана.
4. *Горячая перезагрузка* — возможность изменять интерфейс без перекомпиляции C++ кода.
5. *Встроенный JavaScript* — для простой логики отображения (фильтрация, форматирование дат).

**SQL** используется для работы с базой данных SQLite:

1. *Стандартизированный язык* — SQL-92 совместимость.
2. *Декларативные запросы* — описание «что получить», а не «как получить».
3. *Поддержка транзакций* — атомарность операций (например, удаление группы с каскадным удалением студентов).

### 1.2.4 Информационное обеспечение

**Источники данных:**

Все данные вводятся пользователями через графический интерфейс приложения:
- Учётные данные (логин, пароль, имя, роль) — вводятся администратором при создании аккаунта.
- Данные групп (номер) — вводятся администратором или зам. директора.
- Данные студентов (номер, ФИО, статус) — вводятся вручную или импортируются списком.
- Оценки — вводятся преподавателем в ведомости.

**Организация хранения данных:**

Система использует две базы данных SQLite:

1. **accounts.sqlite** — общая для всех пользователей, хранит учётные записи. Расположение: `%APPDATA%\offline-edm\offline-edm\accounts.sqlite`.

2. **academic.sqlite** — индивидуальная для каждого пользователя, хранит группы, студентов, дисциплины и ведомости. Расположение: `%APPDATA%\offline-edm\offline-edm\users\{login}\academic.sqlite`.

**Структура каталогов:**

```
%APPDATA%\offline-edm\offline-edm\
├── accounts.sqlite           — учётные записи
└── users\
    ├── admin\
    │   └── academic.sqlite   — данные администратора
    ├── ivanov\
    │   └── academic.sqlite   — данные преподавателя Иванова
    └── petrova\
        └── academic.sqlite   — данные зам. директора Петровой
```

**Сторонние сервисы:**

В текущей версии (MVP) сторонние сервисы не используются. Приложение полностью автономно. В будущих версиях планируется добавление серверной части для синхронизации данных между пользователями.

**Анализ аналогов:**

**1С:Колледж** — наиболее функциональная система, однако имеет высокую стоимость лицензии и требует значительных затрат на внедрение и обучение персонала. Требует сервер 1С для работы, что исключает полноценный офлайн-режим.

**Деканат+** — специализированная система для учебных заведений. Не поддерживает офлайн-режим, имеет фиксированный формат ведомостей, который может не соответствовать образцу конкретного колледжа. Отсутствует процесс электронного согласования.

**Moodle** — система управления обучением (LMS), ориентированная на дистанционное образование. Не предназначена для ведения аттестационных ведомостей в формате колледжа. Требует веб-сервер для работы.

**Google Docs** — универсальный инструмент для совместной работы с документами. Не имеет специализированных функций для ведомостей, не поддерживает ролевую модель с процессом согласования, зависит от интернет-подключения.

**Вывод:**

Существующие системы либо имеют высокую стоимость (1С:Колледж), либо не поддерживают полноценный офлайн-режим (Деканат+, Moodle), либо не предназначены для специализированного документооборота ведомостей (Google Docs). Ни одна из рассмотренных систем не воспроизводит точный формат аттестационной ведомости политехнического колледжа НовГУ и не реализует процесс электронного согласования с заместителем директора.

Разработка собственной системы оправдана следующими факторами:
- Полное соответствие формату ведомости колледжа.
- Встроенный процесс согласования (преподаватель → зам. директора).
- Полноценная работа без интернета.
- Нулевая стоимость лицензии.
- Возможность доработки под изменяющиеся требования.
- Защита данных (хеширование паролей PBKDF2-SHA256).
