# 2 Специальная, практическая, опытно-экспериментальная часть

## 2.1 Анализ задачи

### 2.1.1 Информационное моделирование предметной области

Информационная модель предметной области разработана на основе анализа процессов документооборота учебных ведомостей в политехническом колледже. Выделены следующие основные сущности:

**Основные сущности системы:**

1. **Пользователь (User)** — учётная запись сотрудника колледжа с ролью (администратор, преподаватель, заместитель директора).
2. **Учебная группа (StudyGroup)** — группа студентов с уникальным номером (например, 5996 РБ).
3. **Студент (Student)** — обучающийся, привязанный к группе, с ФИО, номером и статусом.
4. **Дисциплина (Discipline)** — учебный предмет (Математика, Программирование и т.д.).
5. **Ведомость (GradeSheet)** — аттестационная ведомость, связывающая группу и дисциплину, содержащая оценки студентов.

**ER-диаграмма базы данных:**

```
┌─────────────────────┐       ┌─────────────────────┐
│       USERS         │       │    STUDY_GROUPS      │
├─────────────────────┤       ├─────────────────────┤
│ login (PK)          │       │ group_id (PK)        │
│ display_name        │       │ number               │
│ salt                │       │ created_at           │
│ password_hash       │       └──────────┬───────────┘
│ created_at          │                  │
│ role                │                  │ 1:N
└─────────────────────┘                  │
                                         ▼
┌─────────────────────┐       ┌─────────────────────┐
│    DISCIPLINES      │       │      STUDENTS        │
├─────────────────────┤       ├─────────────────────┤
│ discipline_id (PK)  │       │ student_id (PK)      │
│ name                │       │ group_id (FK)        │
└──────────┬──────────┘       │ student_number       │
           │                  │ fio                  │
           │ 1:N              │ status               │
           │                  │ grade                │
           ▼                  └─────────────────────┘
┌─────────────────────┐
│    GRADE_SHEETS     │
├─────────────────────┤
│ sheet_id (PK)       │
│ group_id (FK)       │
│ discipline_id (FK)  │
│ title               │
│ created_at          │
│ status              │
│ approved_by         │
└─────────────────────┘
```

**Описание таблиц базы данных:**

Таблица 2.1 — Структура таблицы USERS (accounts.sqlite)

| Поле | Тип | Описание |
|------|-----|----------|
| login | TEXT, PK | Логин пользователя (3–48 символов) |
| display_name | TEXT | Отображаемое имя |
| salt | BLOB | Случайная соль (16 байт) |
| password_hash | BLOB | PBKDF2-SHA256 хеш (120000 итераций) |
| created_at | INTEGER | Дата создания (Unix timestamp, мс) |
| role | TEXT | Роль: admin, teacher, deputy |

Таблица 2.2 — Структура таблицы STUDY_GROUPS (academic.sqlite)

| Поле | Тип | Описание |
|------|-----|----------|
| group_id | TEXT, PK | UUID группы |
| number | TEXT | Номер группы (напр. «5996 РБ») |
| created_at | INTEGER | Дата создания (Unix timestamp, мс) |

Таблица 2.3 — Структура таблицы STUDENTS (academic.sqlite)

| Поле | Тип | Описание |
|------|-----|----------|
| student_id | TEXT, PK | UUID студента |
| group_id | TEXT, FK | Ссылка на группу |
| student_number | TEXT | Номер зачётной книжки |

| Поле | Тип | Описание |
|------|-----|----------|
| discipline_id | TEXT, PK | UUID дисциплины |
| name | TEXT, UNIQUE | Название дисциплины |

Таблица 2.5 — Структура таблицы GRADE_SHEETS (academic.sqlite)

| Поле | Тип | Описание |
|------|-----|----------|
| sheet_id | TEXT, PK | UUID ведомости |
| group_id | TEXT, FK | Ссылка на группу |
| discipline_id | TEXT, FK | Ссылка на дисциплину |
| title | TEXT | Заголовок ведомости |

**Связи между таблицами:**

- STUDY_GROUPS (1) → (N) STUDENTS — одна группа содержит много студентов
- STUDY_GROUPS (1) → (N) GRADE_SHEETS — для одной группы может быть создано несколько ведомостей
- DISCIPLINES (1) → (N) GRADE_SHEETS — одна дисциплина может фигурировать в нескольких ведомостях

**Расположение файлов базы данных:**

```
%APPDATA%\offline-edm\offline-edm\
├── accounts.sqlite              — учётные записи (общая)
└── users\
    └── {login}\
        └── academic.sqlite      — группы, студенты, дисциплины, ведомости
```

### 2.1.2 Проектирование пользовательского интерфейса

Пользовательский интерфейс разработан с использованием декларативного языка QML (Qt Quick) и построен по принципу «боковая панель + основная область».

**Экран 1: Страница авторизации (LoginPage)**

Расположение элементов:
- Логотип и название системы — в верхней части для идентификации приложения
- Поле ввода логина — с иконкой пользователя для визуальной подсказки
- Поле ввода пароля — с кнопкой показа/скрытия пароля
- Кнопка «Войти» — крупная, контрастная, в нижней части формы
- Информационный блок о защищённом соединении — для повышения доверия пользователя

Обоснование: форма авторизации расположена по центру экрана, что соответствует паттерну проектирования для десктопных приложений. Минимальное количество элементов снижает когнитивную нагрузку.

**Экран 2: Главная страница (Main)**

Расположение элементов:
- Левая боковая панель (278 px):
  - Логотип и название системы
  - Индикатор онлайн/офлайн статуса
  - Навигационное меню (Все группы, Мои группы, С ведомостями, Без студентов, На согласование)
  - Кнопка «Управление аккаунтами» (только для администратора)
  - Информация о текущем пользователе с бейджем роли
- Основная область:
  - Заголовок «Учебные группы» с кнопкой создания
  - Поле поиска по номеру группы
  - Список групп в виде карточек

Обоснование: боковая панель обеспечивает постоянный доступ к навигации. Карточный дизайн списка групп позволяет отображать сводную информацию (количество студентов, ведомостей) без перехода на отдельную страницу.

**Экран 3: Страница студентов группы (GroupPage)**

Расположение элементов:
- Заголовок с номером группы и кнопкой «Назад»
- Кнопки «+ Студент» и «Импорт списком»
- Таблица студентов: №, ФИО, Статус, Действия

**Экран 4: Страница дисциплин и ведомостей (DisciplinesPage)**

Расположение элементов:
- Левая колонка: список дисциплин с возможностью добавления и создания ведомости
- Правая колонка: список существующих ведомостей группы

**Экран 5: Аттестационная ведомость (GradeSheetPage)**

Расположение элементов:
- Шапка ведомости (по образцу бумажного документа): название учреждения, номер ведомости, дата, дисциплина, группа
- Таблица: № п/п, ФИО студента, Отметка о допуске, Оценка (редактируемое поле), Подпись, Действия (редактировать/удалить)
- Подвал: итоги оценок, подписи

Обоснование: интерфейс ведомости максимально приближен к бумажному аналогу, что снижает порог вхождения для сотрудников, привыкших к бумажному документообороту.

**Экран 6: Страница согласования (ApprovalPage)**

Расположение элементов:
- Список ведомостей со статусом «Ожидает согласования»
- Для каждой ведомости: кнопки «Просмотр», «Подтвердить», «Отклонить»

## 2.2 Описание логической структуры

### 2.2.1 Алгоритм программы

**Общий алгоритм работы программы:**

1. Запуск приложения → инициализация AuthController → проверка/создание БД accounts.sqlite → создание учётной записи admin (если отсутствует)
2. Отображение страницы авторизации → ввод логина и пароля → вычисление PBKDF2-хеша → сравнение с хранимым хешем
3. При успешной авторизации → открытие сессии (AppController.openSession) → загрузка данных из academic.sqlite
4. Работа с группами, студентами, ведомостями → все изменения сохраняются в SQLite
5. Экспорт ведомости → генерация HTML-файла → сохранение на рабочий стол / открытие для печати

**Алгоритм аутентификации (подробно):**

```
Начало
  │
  ▼
Получить логин и пароль от пользователя
  │
  ▼
Нормализовать логин (toLower, убрать спецсимволы)
  │
  ▼
Запросить из БД: salt, password_hash для данного логина
  │
  ├── Не найден → Вывести «Неверный логин или пароль»
  │
  ▼
Вычислить: trial = PBKDF2-SHA256(пароль, salt, 120000 итераций, 32 байта)
  │
  ▼
Сравнить trial и password_hash (побайтово, constant-time)
  │
  ├── Не совпадает → Вывести «Неверный логин или пароль»
  │
  ▼
Сохранить login, displayName, role в сессии
  │
  ▼
Вызвать AppController.openSession(путь, логин, имя, роль)
  │
  ▼
Загрузить данные пользователя из academic.sqlite
  │
  ▼
Отобразить главную страницу
  │
  ▼
Конец
```

**Алгоритм создания ведомости:**

```
Начало
  │
  ▼
Пользователь выбирает группу и дисциплину
  │
  ▼
Генерация UUID для ведомости
  │
  ▼
Получить номер группы и название дисциплины из БД
  │
  ▼
Сформировать заголовок: «Ведомость: {группа} — {дисциплина}»
  │
  ▼
INSERT INTO grade_sheets (sheet_id, group_id, discipline_id, title, created_at, status='draft')
  │
  ▼
Обновить список ведомостей в UI
  │
  ▼
Открыть страницу ведомости для заполнения оценок
  │
  ▼
Конец
```

**Алгоритм согласования ведомости:**

```
Начало
  │
  ▼
Преподаватель нажимает «На согласование»
  │
  ▼
UPDATE grade_sheets SET status = 'pending_approval' WHERE sheet_id = ?
  │
  ▼
Ведомость появляется во вкладке «На согласование» у зам. директора
  │
  ▼
Зам. директора просматривает ведомость
  │
  ├── «Подтвердить» → UPDATE SET status='approved', approved_by=?
  │
  ├── «Отклонить» → UPDATE SET status='draft'
  │
  ▼
Конец
```

**Алгоритм экспорта ведомости в HTML:**

```
Начало
  │
  ▼
Получить метаданные ведомости из БД (findGradeSheet)
  │
  ▼
Получить список студентов группы (listStudents)
  │
  ▼
Сформировать HTML-документ:
  - Шапка: название учреждения, дисциплина, группа, дата
  - Таблица: № п/п, ФИО, допуск, оценка, подпись
  - Подвал: итоги, подписи
  │
  ▼
Сохранить файл на рабочий стол: {имя}.html
  │
  ▼
Вернуть путь к файлу (или открыть в браузере для печати)
  │
  ▼
Конец
```

### 2.2.2 Используемые методы

**Методы защиты данных:**

1. **Хеширование паролей (PBKDF2-SHA256)** — реализовано в классе AuthController с использованием Qt-функции QPasswordDigestor::deriveKeyPbkdf2. Параметры: 120000 итераций, соль 16 байт, длина ключа 32 байта. Метод slowEqual обеспечивает сравнение хешей за постоянное время (защита от timing-атак).

2. **Разграничение прав доступа** — реализована ролевая модель с тремя ролями:
   - Администратор (admin) — полный доступ, управление учётными записями
   - Преподаватель (teacher) — создание ведомостей, выставление оценок, отправка на согласование
   - Заместитель директора (deputy) — создание ведомостей, согласование/отклонение

3. **Мониторинг сетевого подключения** — используется QNetworkInformation API для отслеживания состояния сети в реальном времени.

**Методы работы с данными:**

1. **Паттерн Repository** — класс AcademicRepository инкапсулирует все SQL-запросы и предоставляет высокоуровневый API для работы с данными.

2. **Модели данных для QML** — классы GroupListModel, StudentListModel, DisciplineListModel, GradeSheetListModel наследуют QAbstractListModel и предоставляют данные для отображения в QML через механизм ролей (roleNames).

3. **Миграции базы данных** — при обновлении схемы используются команды ALTER TABLE с обработкой ошибок (если колонка уже существует, ошибка игнорируется).

4. **Импорт данных** — метод importStudentsFromLines разбирает текст построчно, разделяя поля символом «/», что позволяет быстро добавлять списки студентов.

**Самостоятельно разработанные методы:**

1. Метод exportGradeSheetHtml — генерация HTML-документа аттестационной ведомости по образцу бумажного документа колледжа.
2. Система статусов ведомостей (draft → pending_approval → approved) с контролем переходов.
3. Механизм Q_INVOKABLE для вызова C++ методов из QML-интерфейса.

4. AppController преобразует данные в модели (GroupListModel и т.д.) для отображения в QML.
5. Сигналы (signals) уведомляют QML об изменениях данных для автоматического обновления интерфейса.

## 2.3 Описание работы программы

### 2.3.1 Общие сведения

**Название программы:** ДокументПро (Offline EDM)

**Краткое описание:** Десктопное приложение для автоматизации документооборота учебных ведомостей в политехническом колледже. Обеспечивает создание, заполнение, согласование и печать аттестационных ведомостей с разграничением прав доступа и возможностью работы в офлайн-режиме.

**Версия:** 1.0 (MVP)

**Платформа:** Windows 10/11 (x86_64)

---

### 2.3.2 Функциональное назначение

Программа предназначена для:

1. Управления учебными группами (создание, удаление, просмотр).
2. Ведения списков студентов (добавление, редактирование, импорт, удаление).
3. Управления дисциплинами (добавление новых дисциплин).
4. Создания аттестационных ведомостей по образцу бумажного документа.
5. Выставления оценок студентам непосредственно в ведомости.
6. Отправки ведомостей на согласование заместителю директора.
7. Согласования или отклонения ведомостей.
8. Экспорта ведомостей в HTML-формат для печати.
9. Управления учётными записями пользователей с ролевой моделью.
10. Мониторинга состояния сетевого подключения.

---

### 2.3.3 Связи с другими программами

- **Веб-браузер** — используется для открытия экспортированных HTML-ведомостей и их печати через стандартный диалог печати браузера.
- **Операционная система Windows** — используется API QNetworkInformation для мониторинга сетевого подключения.
- **SQLite** — встроенная СУБД, не требующая отдельного сервера.

### 2.4.3 Выполнение программы

#### 2.4.3.1 Работа пользователя (преподаватель)

1. **Вход в систему:** ввести логин и пароль, нажать «Войти».
2. **Просмотр групп:** на главной странице отображается список учебных групп.
3. **Просмотр студентов:** нажать «Студенты» на карточке группы.
4. **Создание ведомости:** нажать «Ведомости» → выбрать дисциплину → «Создать ведомость».
5. **Заполнение оценок:** в открывшейся ведомости кликнуть на поле «Оценка» напротив студента и ввести цифру (1–5). Оценка сохраняется автоматически при переходе к следующему полю.
6. **Редактирование студента:** нажать иконку «✎» в колонке «Действия», изменить ФИО или статус, нажать «💾» для сохранения.
7. **Отправка на согласование:** нажать кнопку «📤 На согласование» в шапке ведомости.
8. **Печать ведомости:** нажать «🖨 Печать» — ведомость откроется в браузере для печати.
9. **Выход:** нажать «Выйти» в нижней части боковой панели.

#### 2.4.3.2 Работа администратора

Администратор имеет доступ ко всем функциям преподавателя, а также:

1. **Управление аккаунтами:** нажать «⚙ Управление аккаунтами» в боковой панели.
2. **Создание пользователя:** заполнить логин, отображаемое имя, пароль, выбрать роль (Преподаватель / Заместитель директора / Администратор), нажать «Создать».
3. **Создание групп:** нажать «+ Создать группу», ввести номер группы.
4. **Удаление групп:** нажать «✕ Удалить» на карточке группы (удаляются все студенты и ведомости).
5. **Согласование ведомостей:** нажать «На согласование» в боковой панели, просмотреть ведомость, нажать «✓ Подтвердить» или «✕ Отклонить».
6. **Скачивание ведомости:** нажать «⬇ Скачать» — файл сохраняется на рабочий стол в формате HTML.

#### 2.4.3.3 Техника безопасности при работе на компьютере

**Общие требования:**

1. К работе на компьютере допускаются лица, прошедшие инструктаж по технике безопасности.
2. Рабочее место должно быть оборудовано в соответствии с СанПиН 1.2.3685-21.
3. Расстояние от глаз до монитора должно составлять 50–70 см.
4. Непрерывная работа за компьютером не должна превышать 2 часов, после чего необходим перерыв 15 минут.

**Требования перед началом работы:**

1. Убедиться в исправности электрооборудования.
2. Проверить надёжность подключения кабелей.
3. Отрегулировать освещение рабочего места.
4. Отрегулировать высоту стула и положение монитора.

**Требования во время работы:**

1. Не допускать попадания влаги на клавиатуру и системный блок.
2. Не размещать на системном блоке посторонние предметы.
3. При появлении запаха гари или искрения немедленно отключить оборудование от сети.
4. Соблюдать правильную посадку: спина прямая, ноги на полу.

**Требования по окончании работы:**

1. Закрыть все программы и корректно завершить работу операционной системы.
2. Отключить монитор.
3. Привести рабочее место в порядок.
